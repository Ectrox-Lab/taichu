# Qwen 3.5-plus - 代码-1：分布式共识BFT-Raft

## 1. 系统架构与设计思路

将Raft共识算法扩展以容忍拜占庭故障（BFT），基于PBFT的两阶段提交思想，结合Raft的Leader选举和日志复制机制。

**配置**：
- 5个节点，容忍f=1个拜占庭节点
- 所有消息使用数字签名验证身份和完整性
- 法定人数：2f+1 = 3

## 2. 核心数据结构

```go
class Node:
    def __init__(self, node_id, peers, private_key):
        self.id = node_id
        self.peers = peers
        self.private_key = private_key
        self.public_keys = {pid: pub_key for pid in peers}
        
        # 持久化状态
        self.current_term = 0
        self.voted_for = None
        self.log = []  # (term, command)
        
        # 易失性状态
        self.commit_index = 0
        self.last_applied = 0
        
        # Leader状态
        self.next_index = {}
        self.match_index = {}
        
        # 定时器
        self.election_timeout = random(150, 300)  # ms
        self.heartbeat_interval = 50  # ms
```

## 3. Leader选举

### 选举流程

1. **超时触发**：Follower超时未收到心跳则转为Candidate

2. **发起选举**：
   - 增加current_term
   - 设置voted_for = self.id
   - 向所有节点发送RequestVote

3. **投票规则**：
   - 若term > current_term，更新任期并转为Follower
   - 若term == current_term且未投票，检查日志新旧
   - 若候选人日志至少与本地一样新，则投票

4. **计票**：
   - 收集至少3票（2f+1）则当选Leader
   - 收到更高任期响应则降级为Follower

### 防拜占庭措施

- 所有投票需签名，防止伪造
- 拜占庭节点可能虚假投票，但正常节点只投一票
- 最多一个Leader获得多数票（3票）

## 4. 日志复制（两阶段提交）

### 正常流程

**Propose阶段**：
- Leader创建新日志条目(t, cmd)
- 向所有Follower发送Propose消息
- Follower验证签名，检查日志一致性

**Prepare阶段**：
- Follower将条目写入本地日志（未提交）
- 返回Prepare确认（签名）

**Commit阶段**：
- Leader收集至少3个Prepare确认
- 标记条目为已提交，增加commit_index
- 向所有节点发送Commit消息

### 处理拜占庭Leader

- **分叉检测**：Leader对不同Follower发送不同日志，无法获得3个相同Prepare
- **日志一致性检查**：Follower拒绝与前一条目不匹配的提议
- **新Leader选举**：拜占庭Leader被检测后，触发新选举

## 5. 安全性证明

### 证明1：写确认永久有效

- 写确认意味着Leader收到3个Prepare（包括自己）
- 该条目存在于至少3个节点日志中
- 新Leader必须获得3票，至少包含2个正常节点
- 正常节点保存已确认条目，新Leader日志必然包含该条目
- **结论**：已提交条目永久有效

### 证明2：读一致性

- 客户端向至少3个节点发送读请求
- 取多数一致的值（至少2个相同）
- 任何两个多数集必有交集
- **结论**：读到的值唯一且一致

### 证明3：系统活性

- 随机超时确保总有正常节点发起选举
- 拜占庭Leader可被检测并替换
- 只要网络最终同步，系统保证进展
- **结论**：在部分同步假设下，系统具有活性

```python
def become_leader(self):
    self.role = 'leader'
    for peer in self.peers:
        self.next_index[peer] = len(self.log)
        self.match_index[peer] = -1
    self.broadcast_heartbeat()

def handle_propose(self, msg):
    if msg.term < self.current_term:
        return
    # 检查日志一致性
    if msg.prev_log_index >= len(self.log) or \
       (msg.prev_log_index >= 0 and self.log[msg.prev_log_index][0] != msg.prev_log_term):
        return  # 拒绝
    # 追加日志
    for entry in msg.entries:
        self.log.append(entry)
    # 返回Prepare确认
    self.send(msg.leader_id, 'PrepareResponse', ...)

def handle_prepare_response(self, msg):
    if msg.success:
        self.match_index[msg.sender] = msg.index
        count = 1  # 自己
        for peer in self.peers:
            if self.match_index.get(peer, -1) >= msg.index:
                count += 1
        if count >= 3:  # 2f+1
            self.commit_index = msg.index
            self.broadcast_commit()
```

## 8. 复杂度分析

- **消息复杂度**：每写请求约15条消息（Propose + Prepare + Commit）
- **时间复杂度**：2轮RTT（约20-40ms）
- **空间复杂度**：O(log长度) 每节点

## 9. 总结

本实现通过数字签名和两阶段提交，将Raft从CFT升级为BFT，确保在1个拜占庭节点存在时，系统仍保持安全性和活性。
